# Подробная инструкция по развёртыванию на Vercel

Пошаговое руководство по публикации проекта «kyrsovaya» (аренда видеооборудования) на облачном хостинге Vercel. После выполнения инструкции сайт и API (включая нейросеть GigaChat) будут работать в интернете по одному адресу.

---

## Что уже подготовлено в проекте

- **Фронтенд**: Vite + React, собирается командой `npm run build`, результат в папке `dist`.
- **API**: папка `api/` с serverless-функциями Vercel:
  - `GET /api/health` — проверка работы сервера;
  - `GET /api/debug-models` — список моделей GigaChat (для отладки);
  - `POST /api/suggest-tariffs` — подбор сценариев по запросу пользователя (нейросеть).
- **Конфигурация**: файл `vercel.json` в корне проекта задаёт сборку и маршрутизацию для SPA.

Локально для разработки по-прежнему можно использовать `npm run dev` и при необходимости старый сервер из папки `server/`. Для Vercel он не используется.

---

## Шаг 1. Регистрация и установка Vercel CLI (по желанию)

### 1.1. Аккаунт Vercel

1. Откройте в браузере: [https://vercel.com](https://vercel.com).
2. Нажмите **Sign Up**.
3. Войдите через **GitHub**, **GitLab** или **Bitbucket** (удобнее всего GitHub — деплой по коммиту в репозиторий).
4. При первом входе разрешите Vercel доступ к вашим репозиториям (можно только выбранный репозиторий).

### 1.2. Установка Vercel CLI (необязательно)

CLI нужен только если хотите деплоить из терминала без привязки к Git.

- **Windows (PowerShell от имени администратора):**
  ```powershell
  npm install -g vercel
  ```
- После установки выполните в терминале:
  ```bash
  vercel login
  ```
  и войдите в аккаунт по ссылке из браузера.

Если планируете деплоить только через сайт Vercel и GitHub — CLI можно не ставить.

## Шаг 2. Подготовка репозитория на GitHub

Чтобы Vercel мог автоматически собирать и обновлять проект, код должен быть в Git-репозитории (лучше всего на GitHub).

### 2.1. Инициализация Git (если ещё не сделано)

В корне проекта (папка `kyrsovaya`, где лежат `package.json` и `vercel.json`):

```bash
cd kyrsovaya
git init
```

### 2.2. Файл `.gitignore`

Убедитесь, что в корне есть `.gitignore` и в нём указано хотя бы:

```
node_modules
dist
.env
.env.local
.env.*.local
.vercel
```

- `node_modules` и `dist` не должны попадать в репозиторий.
- Файлы `.env` с ключами **не загружайте** в Git — секреты задаются только в настройках Vercel (шаг 4).

### 2.3. Первый коммит и отправка на GitHub

1. Создайте новый репозиторий на [https://github.com/new](https://github.com/new) (например, имя `kyrsovaya`), без README и без .gitignore.
2. В папке проекта выполните:

```bash
git add .
git commit -m "Prepare for Vercel deployment"
git branch -M main
git remote add origin https://github.com/ВАШ_ЛОГИН/kyrsovaya.git
git push -u origin main
```

Подставьте вместо `ВАШ_ЛОГИН` свой логин GitHub и при необходимости измените имя репозитория.

## Шаг 3. Создание проекта в Vercel

### 3.1. Импорт репозитория

1. Зайдите на [https://vercel.com](https://vercel.com) и нажмите **Add New…** → **Project**.
2. В списке репозиториев выберите **kyrsovaya** (или как вы назвали репозиторий).
3. Если репозитория нет — нажмите **Import Git Repository** и при необходимости снова подключите GitHub/GitLab.

### 3.2. Настройки сборки (Build and Output Settings)

Vercel обычно сам определяет Vite, но лучше проверить:

| Поле | Значение |
|------|----------|
| **Framework Preset** | Vite (или Other, если Vite не предлагается) |
| **Root Directory** | оставить пустым (корень репозитория = папка с `package.json` и `vercel.json`) |
| **Build Command** | `npm run build` (или оставить по умолчанию) |
| **Output Directory** | `dist` |
| **Install Command** | `npm install` |

Если в корне репозитория лежит не сама папка `kyrsovaya`, а родительская папка с нею внутри, тогда в **Root Directory** укажите `kyrsovaya`, чтобы корнем проекта была именно она.

### 3.3. Переменные окружения (пока можно не заполнять)

На этом же экране есть блок **Environment Variables**. Пока нажмите **Deploy** — первый деплой можно сделать без ключей GigaChat. Сайт откроется, но подбор сценариев по нейросети будет выдавать ошибку «Нейросеть недоступна», пока не добавлены переменные (шаг 4).

## Шаг 4. Добавление переменных окружения (ключи GigaChat)

Чтобы работал подбор тарифов через GigaChat, нужно передать в Vercel те же переменные, что и в локальном `.env`.

### 4.1. Где взять ключи

1. Зарегистрируйтесь в [GigaChat для разработчиков](https://developers.sber.ru/studio) (кабинет Sber).
2. Создайте приложение и получите **Client ID** и **Client Secret** (или инструкцию по их получению).
3. Локально у вас может быть файл `server/.env` с содержимым вида:
   ```env
   GIGACHAT_CLIENT_ID=ваш_client_id
   GIGACHAT_CLIENT_SECRET=ваш_client_secret
   ```
   Эти значения и нужно перенести в Vercel.

### 4.2. Добавление в Vercel

1. В Vercel откройте свой проект.
2. Вверху перейдите на вкладку **Settings**.
3. В левом меню выберите **Environment Variables**.
4. Добавьте переменные по одной:

| Name | Value | Среда (Environment) |
|------|--------|----------------------|
| `GIGACHAT_CLIENT_ID` | вставьте ваш Client ID | Production, Preview, Development (можно отметить все три) |
| `GIGACHAT_CLIENT_SECRET` | вставьте ваш Client Secret | Production, Preview, Development |

- Значение **Value** при желании можно скрыть (глазик).
- Кнопка **Save** после каждой переменной.

### 4.3. Вариант через одну переменную (опционально)

Если у вас один ключ в формате base64 `id:secret`:

| Name | Value |
|------|--------|
| `GIGACHAT_CREDENTIALS` | строка в base64 вида `client_id:client_secret` |

Тогда `GIGACHAT_CLIENT_ID` и `GIGACHAT_CLIENT_SECRET` можно не задавать — код в `api/suggest-tariffs.js` и `api/_shared.js` умеет читать credentials из этой переменной.

### 4.4. Применение переменных

После сохранения переменных нужно заново задеплоить проект, чтобы они подхватились:

- Либо сделайте любой новый коммит и пуш в репозиторий — Vercel задеплоит автоматически.
- Либо на вкладке **Deployments** откройте последний деплой → три точки **⋯** → **Redeploy** (без изменения кода, переменные подтянутся из настроек).

## Шаг 5. Деплой и проверка

### 5.1. Первый деплой

- Если вы на шаге 3 нажали **Deploy**, дождитесь окончания сборки (обычно 1–2 минуты).
- Если деплой ещё не запускали: в проекте нажмите **Deployments** → **Deploy** (или снова **Add New…** → **Project** и выберите тот же репозиторий с уже заданными настройками).

По окончании сборки Vercel покажет ссылку вида:

`https://kyrsovaya-xxxxx.vercel.app`

(или ваш кастомный домен, если вы его уже настроили).

### 5.2. Что проверить

1. **Главная страница**  
   Откройте в браузере ссылку проекта. Должна открыться главная страница приложения (поиск, карточки сценариев и т.д.).

2. **Проверка API**  
   Откройте в новой вкладке:
   ```text
   https://ВАШ_ПРОЕКТ.vercel.app/api/health
   ```
   Ожидаемый ответ в формате JSON:
   ```json
   {"ok":true,"message":"Сервер запущен"}
   ```

3. **Подбор сценариев (нейросеть)**  
   - Зайдите на главную.
   - Введите в поиск, например, «свадьба» или «стрим».
   - Нажмите поиск.  
   Если переменные GigaChat заданы и ключи валидны, появятся подсказанные сценарии. Если нет — будет сообщение вроде «Нейросеть недоступна»; тогда проверьте шаг 4 и сделайте **Redeploy**.

4. **Остальные страницы**  
   Проверьте переходы по разделам: О компании, Поддержка, Сценарии, Конструктор. Все они должны открываться без 404 (благодаря SPA и `vercel.json`).

---

## Шаг 6. Дальнейшие обновления

- Любой пуш в ветку, подключённую к проекту (обычно `main`), запускает новый деплой.
- Просмотр логов: **Deployments** → выберите деплой → **Building** / **Functions** (логи сборки и serverless-функций).
- Откат: в **Deployments** выберите более ранний успешный деплой → **⋯** → **Promote to Production**.

## Возможные проблемы и решения

### Сборка падает с ошибкой

- Убедитесь, что в репозитории есть `package.json` в корне (или в Root Directory, который вы указали).
- В логах сборки посмотрите шаг (`npm install` или `npm run build`), на котором ошибка. Часто помогает локально выполнить `npm ci` и `npm run build` в той же папке.

### «Нейросеть недоступна» при подборе тарифов

- Проверьте, что в **Settings** → **Environment Variables** заданы `GIGACHAT_CLIENT_ID` и `GIGACHAT_CLIENT_SECRET` (или `GIGACHAT_CREDENTIALS`).
- После добавления/изменения переменных сделайте **Redeploy** деплоя.

### Страницы дают 404 при прямом переходе по ссылке

- Должен быть один rewrite в `vercel.json`, отправляющий запросы на `/index.html`. Проверьте, что файл `vercel.json` в корне проекта и не перезаписан другими настройками в Vercel (в **Settings** → **General** не должно быть конфликтующих правил).

### API отвечает 500

- Откройте **Deployments** → последний деплой → **Functions** → выберите функцию (например, `api/suggest-tariffs`) и посмотрите логи ошибок. По ним можно понять, не падает ли загрузка `equipment.json` или запрос к GigaChat.

### Файл `equipment.json` не найден

- В коде используется путь `process.cwd() + '/src/data/equipment.json'`. Корнем на Vercel считается корень репозитория (или Root Directory). Убедитесь, что в репозитории есть папка `src/data/` и в ней файл `equipment.json`.
